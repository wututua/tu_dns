## 1. Go 开发规范

### 1.1 代码风格

- **强制** `gofmt`：所有 `.go` 文件提交前必须经过 `gofmt` 格式化。CI 会检查 `gofmt -l .` 结果是否为空。
- **禁止** `goimports` 自动重排以外的非标准 import 分组。标准库、第三方、本地包之间空行分隔。
- **命名**：
  - 包名小写单数（`config`、`domain`、`record`），不使用连接线或下划线。
  - 导出类型/函数使用驼峰（`SaveInput`、`NewService`）。
  - 私有变量保持简洁，但避免单字母（循环索引除外）。
- **行宽不做硬限制**，但建议不超过 120 字符。

### 1.2 包结构

- 遵循 flat 布局（无 `cmd/`、`internal/` 嵌套），业务包直接位于根目录。
- 每个包只做一件事。如果包描述出现"和"，考虑拆分。
- 禁止循环依赖。如果包 A 和包 B 互相引用，提取公共接口到新包。
- `models/` 是唯一持有 GORM 模型定义的地方。其他包引用 `models.XXX`，不自行定义数据库结构。

### 1.3 错误处理

- **总是检查错误**。忽略错误必须用 `_ =` 显式标记并注释原因。
- 业务错误使用 `errors.New` 或 `fmt.Errorf`，不使用 `panic`/`recover`（Gin Recovery 中间件除外）。
- Service 层返回描述性错误，Handler 层将其转为 HTTP 响应。错误信息不应暴露内部实现细节。

```go
// Bad: 暴露内部
return nil, fmt.Errorf("db query failed: %w", err)

// Good: 用户友好
return nil, errors.New("用户名已存在")
```

### 1.4 测试

- 测试文件以 `_test.go` 结尾，与源码同包。
- 单元测试不依赖外部数据库或网络。需要数据库的测试使用 SQLite 内存模式。
- 表驱动测试优先。
- `go test ./... -count=1` 必须在提交前通过。

### 1.5 并发

- 使用 `sync.RWMutex` 保护包级变量（参考 `config/config.go`）。
- 初始化在 `main.go` 或服务构造函数中完成，避免 `init()` 函数。
- GORM 连接池参数（`SetMaxOpenConns`、`SetMaxIdleConns`）在数据库初始化时设置，handler/service 不管理连接。

## 2. 前端开发规范

### 2.1 技术选型

- 框架：React 19 + TypeScript + Vite
- UI 库：HeroUI v3（`@heroui/react`）
- 样式：Tailwind CSS v4 + `tw-animate-css`
- 路由：`react-router-dom` v7
- 不使用外部状态管理库。页面级状态使用 React hooks，跨页状态通过 `auth.tsx` 中的 Context 传递。

### 2.2 组件约定

- 组件文件使用 `.tsx` 扩展名，纯逻辑文件使用 `.ts`。
- 组件名为 PascalCase，文件名与组件名一致（`Dashboard.tsx` → `export function Dashboard`）。
- 页面组件放在 `src/pages/`，通用组件放在 `src/components/`。
- 管理后台页面放在 `src/pages/admin/` 下。
- 禁止 `any` 类型。API 响应类型在 `src/lib/api.ts` 中定义。

### 2.3 前后端交互

- 所有 API 请求通过 `src/lib/api.ts` 中的封装函数发起。
- Token 管理在 `src/lib/auth.tsx` 中统一处理。
- 前端开发服务器代理 `/api` 到后端 `127.0.0.1:8080`（配置在 `vite.config.ts`）。
- 错误处理由各页面自行管理，但没有全局加载/空状态统一组件。

## 3. API 设计规范

### 3.1 响应格式

所有 JSON 接口统一返回：

```json
{"code": 0, "message": "ok", "data": {}}
```

- `code === 0`：成功
- `code !== 0`：业务错误，`message` 描述原因
- `data`：成功时携带数据，失败时省略

### 3.2 HTTP 状态码

| 状态码 | 使用场景 |
|--------|----------|
| 200 | 成功 |
| 400 | 参数验证失败、业务规则冲突 |
| 401 | 未登录或 Token 无效/过期 |
| 403 | 用户被禁用或权限不足 |
| 404 | 资源不存在 |
| 500 | 服务端异常 |
| 503 | 系统未安装或数据库不可用 |

### 3.3 路由命名

- 公开接口：`GET /healthz`、`GET /api/dns/providers`
- 认证接口：`POST /api/auth/login`、`PUT /api/auth/password`
- 资源接口：`GET /api/subdomains`、`POST /api/records`
- 管理接口：`/api/admin/users`、`/api/admin/settings`
- 安装接口：`/api/install/status`、`POST /api/install`
- 支付接口：`/api/pay/alipay/create`、`POST /api/pay/alipay/notify`

### 3.4 新增接口流程

1. 在 `server/router.go` 注册路由。
2. 在对应 handler 文件中（`handlers_*.go`）添加处理器方法。
3. 调用 Service 层执行业务逻辑。
4. 通过 `OK()` / `BadRequest()` / `ServerError()` 响应。
5. 更新 `docs/api.md` 文档。

## 4. 数据库规范

### 4.1 模型变更

- 所有模型定义在 `models/models.go` 中。
- GORM `AutoMigrate` 在启动时自动应用变更，不做版本化 SQL migration。
- **禁止**删除列或修改已有列的类型。如需移除，先标记废弃，两个版本后再清理。
- 新增模型后须加入 `AllModels()` 函数。

### 4.2 查询

- 使用 GORM 链式 API，禁止原始 SQL 字符串拼接。
- 涉及用户数据的查询必须附带 `UserID` 条件，防止越权。
- 分页统一使用 `page` 查询参数，每页固定 20 条。

### 4.3 事务

- 涉及扣费和写入的操作必须在 GORM 事务中执行（参考 `record/service.go`）。
- 第三方 DNS API 不参与事务。上游成功但本地失败时需记录日志等待人工对账。

## 5. Git 工作流

### 5.1 分支策略

- `main`：生产就绪状态，必须通过 CI。
- 功能开发在 feature branch 上完成，提交 PR 合并到 `main`。

### 5.2 Commit 消息

```
<type>: <简短描述>

<可选的详细说明>
```

类型前缀：

| 前缀 | 使用场景 |
|------|----------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `refactor` | 重构 |
| `docs` | 文档变更 |
| `chore` | 构建/CI/工具 |
| `test` | 测试 |
| `style` | 代码格式 |

示例：

```
feat(record): 添加批量删除 DNS 记录接口

fix: 安装向导中 SQLite 路径处理错误

refactor(server): 拆分 router.go 处理器到独立文件
```

### 5.3 PR 要求

- CI 必须绿色。
- 禁止 force push 到 `main`。
- PR 标题使用中文或英文均可，但需说明变更意图。

## 6. 安全规范

### 6.1 红线（硬性禁止）

- **禁止**日志或响应中输出密钥、Token、DSN。
- **禁止**字符串拼接 SQL。
- **禁止**将 `config.yaml`、`data/` 提交到 Git。
- **禁止**使用 `encoding/json` 以外的非参数化用户输入传给数据库。
- **禁止**暴露模拟支付端点到生产环境。

### 6.2 密钥管理

- `security.secret_key` 同时用于 JWT 签发和 DNS Provider 配置加密。
- 更换密钥前必须先迁移所有已加密的 Provider 配置。
- 支付宝私钥存储在数据库 `settings` 表中，管理 API 返回时以 `***` 遮蔽。

### 6.3 输入验证

- 所有用户输入在 Service 层验证，Handler 层只做 JSON 解析。
- 密码最低长度 6，用户名最低长度 3。
- IP 地址、邮箱格式在 Service 层校验。

## 7. 文档规范

### 7.1 文档位置

| 类型 | 位置 |
|------|------|
| 项目级文档 | 根目录（README、LICENSE、CHANGELOG） |
| 技术深度文档 | `docs/` 目录 |
| 开发规程（内部） | `.opencode/` 目录（不提交 GitHub） |
| API 文档 | `docs/api.md` |

### 7.2 文档更新时机

- 新增/修改 API 端点 → 更新 `docs/api.md`。
- 新增/修改 DNS Provider → 更新 `docs/dns-providers.md`。
- 新增/修改配置项 → 更新 `docs/env-vars.md`。
- 架构变更 → 更新 `docs/architecture.md`。
- 部署流程变化 → 更新 `docs/deploy.md`。

### 7.3 跨文档链接

- 使用相对路径引用同一仓库内的文档。
- 迁移文件位置时，必须更新所有引用该文件的链接。
